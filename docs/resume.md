프론트엔드 개발부터 UI/UX 설계, 팀 리딩까지 전 과정을 주도하며 심미성과 기능성을 모두 갖춘 결과물을 만드는 것에 가치를 둡니다. 최근 500만 유저 앱 '데이트팝'의 신규 웹 서비스 개발에 참여하여, 초기 FSD 아키텍처를 재설계하고 데이터 스키마를 시각화(DBML)하여 개발 생산성을 높이는 등 시스템적 문제 해결 역량을 길러왔습니다.

또한 부트캠프 과정에서 팀장으로서 체험 예약 플랫폼을 개발하며 사용자 탐색 단계를 3단계에서 1단계로 단축시키고, Three.js를 이용한 인터랙션을 개발함과 동시에 렌더링 성능을 80% 개선하는 성과를 만들었습니다. 저는 코드와 디자인의 경계에서 소통하며, 기술적 한계를 창의적인 문제 해결로 돌파할 때 최고의 결과물이 나온다고 믿습니다. 앞으로도 협업을 통해 사용자의 기억에 남는 경험을, 가장 안정적인 구조 위에서 구현하는 프론트엔드 개발자가 되고 싶습니다.

## 프로젝트 1/4
데이트팝 체험단 신규 서비스 구축 및 고도화

2025년 11월 10일 - 현재 | 사내 프로젝트 | 역할: 프론트엔드 인턴

누적 500만 다운로드 앱 '데이트팝'의 신규 비즈니스 모델인 체험단 플랫폼의 웹 프론트엔드 개발을 담당했습니다. FSD 아키텍처 구조 개선부터 핵심 도메인 모델링, 디자인 시스템 정립 등 유지보수성과 개발 효율성을 높이는 데 주력했습니다.

1. FSD 아키텍처 구조 개선 및 데이터 스키마 시각화
    
    - 문제점: 초기 도입된 FSD(Feature-Sliced Design) 아키텍처 내에서 잘못된 계층 의존성과 모호한 디렉토리 구조로 개발 혼선 발생.
    - 해결 과정: features, entities, shared 등 계층별 역할을 재정의하고, 모듈 간 순환 참조 및 의존성 규칙을 위반하는 코드를 리팩토링하여 구조적 기틀을 바로잡음. 또한, DB 구조 파악이 어려운 문제를 해결하기 위해 DBML(Database Markup Language)을 활용, 전체 데이터베이스 스키마를 다이어그램으로 시각화하여 공유함.
    - 결과: 아키텍처의 명확성을 높여 협업 효율을 개선하고, 백엔드-프론트엔드 간 데이터 구조에 대한 이해도를 일치시킴.
    
2. 체험단 도메인 모델링 및 UI 컴포넌트 고도화
    
    - 역할: 서비스의 핵심인 체험단 도메인의 데이터 모델을 정의하고, 상태별 카드 및 리스트 UI 구현을 전담했습니다.
    - 성과 - 정교한 모델링: Campaign, CampaignDetail 등 핵심 엔티티 타입을 비즈니스 로직에 맞춰 설계하고 Mock 데이터를 구축함.
    - 성과 - 상태별 UI 대응: 모집 전/중/마감 등 캠페인 생명주기에 따른 다양한 Card Variants와 Tab UI를 구현하고, 탭을 통한 스크롤 인터랙션 등 디테일한 UX를 완성함.
    
3. 타입 시스템 안정화 및 데이터 매핑 표준화
    
    - 문제점: 잦은 기획 변경으로 인해 타입 정의와 실제 요구 데이터 간 불일치가 발생했고, 부정확한 타입 정의로 인해 런타임 에러 위험이 있었음.
    - 해결 과정: 주도적으로 전체 프로젝트의 타입을 점검하여 Campaign, Application 등 핵심 엔티티 타입을 전면 재정의함. 특히 DB의 Snake Case 컬럼명을 프론트엔드의 Camel Case 컨벤션에 맞춰 변경하는 작업을 수행하고, 사용하지 않는 레거시 컴포넌트와 불필요한 테스트 코드를 제거함.
    - 결과: 타입 안정성을 확보하여 런타임 에러를 방지, 일관된 코딩 컨벤션을 확립하여 코드 품질을 향상시킴.
    
4. 디자인 시스템 구축 및 Design Ops 수행
    
    - 문제점: 디자인 퀄리티를 사수하고 개발 생산성을 높이는 공통 컴포넌트 및 스타일 시스템을 구축함.
    - 성과 - 시스템화: z-index 레벨을 설계하고 상수화하여 관리, 각종 재사용 컴포넌트를 구현함.
    - 성과 - 협업 효율화: Figma 디자인 의도와 실제 구현 간의 간극을 좁히기 위해 디자인 피드백을 역제안하고, 일관된 UI 구현을 위해 팀원들과 적극 소통함.
    - 결과: 디자인과 개발 간의 커뮤니케이션 비용을 줄이고, 서비스 전체의 UI 일관성을 확보하여 개발 속도 증대에 기여함.

## 프로젝트 2/4
어디가냥 전국 체험 예약 플랫폼

2025년 5월 27일 ~ 6월 26일 (약 4주) | 팀프로젝트 FE 5명 | 역할: 팀장, 프론트엔드, 프로덕트 디자인

Web: meowhere.vercel.app | Github: github.com/sngminn/Meowhere-Deploy

1. Three.js 기반 3D 시스템 설계 및 최적화 (https://url.kr/i8batk)
    
    - 문제점: 초기 3D 모델의 과도한 용량(5MB 이상)으로 인해, 모바일 환경에서 렌더링 시간이 길어지고 사용자 경험을 저해하는 심각한 성능 저하가 발생함.
    - 해결 과정: 3D 프로그램 Blender에서 압축기능으로 폴리곤을 80% 이상 감축, 용량을 크게 줄임. Three.js Editor에서 웹 환경에 맞게 재질과 조명을 재설계하는 등 단계별 최적화를 직접 수행함.
    - 결과: 3D 에셋 용량을 800KB 미만으로 경량화하고 평균 렌더링 시간을 단축하여, 모바일 환경에서도 안정적인 3D 인터랙션 경험을 확보함.
    
2. 백엔드 제약 극복을 위한 데이터 캐싱 전략 수립 (https://url.kr/ppo3n3)
    
    - 문제점: 가격 데이터 전용 API 부재로, 불필요한 전체 데이터를 매번 호출해야 하는 비효율 발생.
    - 해결 과정: 별도 API 구축이 불가능한 환경에서, Next.js의 unstable_cache를 도입함. 주기적으로 전체 데이터를 가공, 캐싱하는 방식으로 해결함. 통계적 이상치 제거 로직을 추가, UX를 개선함.
    - 결과: 불필요한 데이터 fetching을 최소화하여 서버 부하를 줄였고, 사용자에게는 거의 실시간에 가까운 빠른 데이터 탐색 경험을 제공함.
    
3. 파편화된 사용자 경험 통합 및 지식 전파 (https://url.kr/z7xnww)
    
    - 문제점: 여러 페이지에 필터 기능이 분산되어 있어, 사용자가 원하는 정보를 찾기 위해 평균 3단계의 복잡한 탐색 과정을 거쳐야 했음.
    - 해결 과정: Airbnb 등 주요 서비스의 UX 패턴을 분석하여, 모든 필터 기능을 단일 GNB/BNB 컴포넌트로 통합하는 구조를 직접 설계 및 구현함.
    - 결과: 사용자의 핵심 정보 탐색 단계를 1단계로 단축시켰으며, 해당 컴포넌트의 재사용성을 높이기 위해 팀원들을 위한 Wiki 가이드를 직접 작성 및 배포하여 팀 전체의 개발 생산성 향상에 기여함.
    
4. 디자인 시스템 구축 및 워크플로우 최적화 (https://url.kr/tmigig)
    
    - 문제점: 디자인과 개발 간의 워크플로우 비효율 및 UI 비일관성 문제가 발생함(CSS 계층 충돌, 일관성 없는 애니메이션 등).
    - 해결 과정: 다각적인 디자인 시스템을 직접 구축하여 문제를 해결함. Z-Index 가이드로 CSS 계층 충돌을 방지했고, 모션그래픽 경험을 바탕으로 목적별 Cubic Bezier을 정의했으며, Figma 컴포넌트 구조를 재설계하여 워크플로우를 최적화함. 상태에 따라 SVG내 path 데이터만 조건부로 렌더링하는 단일 SVG 컴포넌트를 설계함.
    - 결과: 디자인과 개발 간의 커뮤니케이션 비용을 줄이고, 서비스 전체의 UI 일관성을 확보하여 개발 속도 증대에 기여함.
    
5. AI를 활용한 협업 프로세스 개선 (https://url.kr/c55cp6)
    
    - 문제점: 코드 리뷰 과정에서 사소한 컨벤션 위반이나 오타 등 반복적인 휴먼 에러를 수정하는 데 불필요한 커뮤니케이션 비용이 발생함.
    - 해결 과정: 협업 효율을 높이기 위해, GitHub의 Gemini Code Assist를 실험적으로 도입하고 팀의 코드 컨벤션을 학습시켜 1차 코드 리뷰를 자동화함.
    - 결과: 단순 피드백 과정을 자동화함으로써, 팀원들이 핵심 비즈니스 로직 리뷰에 더 집중할 수 있는 환경을 조성하고 협업의 질을 향상시킴.

## 프로젝트 3/4
FANDOM-K K-POP 아이돌 투표 및 후원 플랫폼

2025년 3월 6일 ~ 3월 20일 (약 2주) | 팀프로젝트-FE 4명 | 역할: 프론트엔드, 프로덕트 디자인

Web: fandom-k-eight.vercel.app | Github: github.com/sngminn/Fandom-K

1. 랜딩페이지 렌더링 성능 최적화 (https://url.kr/mida5z)
    
    - 문제점: 다수의 스크롤 애니메이션으로 구성된 랜딩페이지에서, 과도한 scroll 이벤트 콜백으로 인해 애니메이션 처리 속도가 4.45초에 달하는 등 심각한 성능 저하가 발생함.
    - 해결 과정: 초기 throttle 및 GPU 가속 적용의 한계를 넘어, React 환경에 최적화된 Framer Motion을 도입함. useScroll과 useTransform 훅을 활용하여 스크롤 진행도에 따른 애니메이션 로직을 선언적으로 관리하고, 렌더링을 16ms (60fps) 주기로 제어함.
    - 결과: 이 최적화 과정을 통해 애니메이션 처리 속도를 4.45초에서 0.47초로 약 9.47배 향상시켰으며, 모든 디바이스에서 60fps의 안정적이고 부드러운 사용자 경험을 확보함.
    
2. 확장성을 고려한 프론트엔드 아키텍처 설계 (https://url.kr/Boxugx)
    
    - 문제점: 프로젝트 전반에 걸쳐 모달, 타이포그래피 등 공통 UI 요소에 대한 규칙이 없어 props drilling, 코드 중복, UI 비일관성 등 유지보수성을 저해하는 문제가 발생함.
    - 해결 과정: Context API를 기반으로 openModal, closeModal을 통해 모든 모달을 제어하는 중앙집중식 시스템을 직접 설계함. 또한, Styled Components의 Theme Provider와 연동되는 체계적인 디자인 시스템(Typography, Color)을 명확한 네이밍 컨벤션(b24 등)과 함께 구축하여 팀 전체에 배포함.
    - 결과: props drilling을 원천적으로 해결하고 새로운 모달 타입을 빠르게 추가할 수 있는 확장 가능한 구조를 구현함. 체계적인 디자인 시스템 도입으로 코드 중복을 줄이고 UI 일관성을 확보하여 팀의 개발 효율을 크게 향상시킴.
    
3. 불안정성에 대응하는 데이터 관리 전략 수립 (https://url.kr/hxlx6s)
    
    - 문제점: 제공된 백엔드 API 서버가 매우 불안정하여, 요청 실패 및 데이터 로딩 지연이 빈번하게 발생, 사용자 경험을 심각하게 저해함.

    - 결과: 불안정한 네트워크 환경에서도 데이터 로딩의 안정성을 확보했으며, 재방문 시 체감 로딩 속도를 크게 개선하고 오프라인에서도 기본 기능을 유지할 수 있도록 함.

1. 프론트엔드 아키텍처: 인증 및 상태 관리 시스템 설계 (https://url.kr/cp1sr6)
    
    - 문제점: 3주라는 짧은 개발 기간 내에 자체 로그인과 안전한 인증 시스템을 구축해야 했으며, 페이지 새로고침 후에도 로그인 상태를 유지하고 여러 컴포넌트에서 사용자 데이터를 공유하기 위한 효율적인 전역 상태 관리 전략이 필요했음.
    - 해결 과정: 자체 JWT 구현 방식의 보안 및 개발 시간 리스크를 고려하여, 표준화된 소셜 로그인 Provider를 제공하고 보안이 검증된 NextAuth.js를 최종 선택함. 또한, Redux 대비 보일러플레이트가 거의 없고 persist 미들웨어로 LocalStorage 연동이 간편한 Zustand를 채택하여 개발 생산성을 극대화함.
    - 결과: NextAuth.js 도입으로 인증 시스템 구축 시간을 단축하여 팀이 핵심 기능 개발에 더 집중할 수 있었으며, Zustand를 통해 상태 관리의 효율성을 높임.
    
2. 협업 및 버전 관리: Git 히스토리 충돌 해결 (https://url.kr/55w1tq)
    
    - 문제점: 팀원의 작업 오류로 인해 공유 브랜치(develop)의 HEAD가 이전 커밋으로 덮어씌워져, 최신 기능 코드 일부가 유실되는 긴급 상황이 발생함.
    - 해결 과정: 다른 팀원의 작업을 유실시킬 수 있는 git reset --hard의 위험성을 인지하고, 팀 회의를 주도하여 히스토리를 보존하는 가장 안전한 방식인 git revert를 통한 복구를 제안 및 최종 해결함.
    - 결과: 이슈 발생 후 30분 이내에 모든 코드를 손실 없이 복구하여 프로젝트 지연을 방지함. 이 사건을 계기로 팀 내 재발 방지를 위한 Git 가이드라인을 공유하여 협업 안정성을 강화함.
    
3. UI/UX 및 최적화: 사용자 경험 디테일 개선 (https://url.kr/cef5g2)
    
    - 문제점: 서비스의 첫인상을 결정하는 랜딩 페이지의 경험, 사소하지만 반복적인 불편함을 유발하는 UI, 그리고 불필요한 리소스로 인한 성능 저하 등 개선이 필요했음.
    - 해결 과정: Matter.js의 물리 엔진과 직접 제작한 Lottie 애니메이션을 결합한 인터랙티브 랜딩 페이지를 단독 개발하여 첫인상을 개선함. 또한, 스크롤 시 GNB에 검색창을 동적으로 노출시켜 UI 접근성을 향상시키고, Figma 프로젝트 리소스를 최적화하여 불필요한 웹 폰트를 41% 제거(약 1.7MB 절감)함.
    - 결과: 동적인 경험과 작은 UX 개선으로 서비스의 전체적인 완성도를 높였으며, 리소스 최적화를 통해 실질적인 초기 로딩 성능 향상을 이뤄냄.
    
4. 기술 스택 트레이드오프: Styled Components와 SSR의 상충 관계 (https://url.kr/yrrwlw)
    
    - 문제점: 팀에 익숙한 Styled Components가 Next.js의 SSR 환경에서 초기 로딩 시 스타일이 적용되지 않는 깜빡임(FOUC) 현상을 유발함.
    - 해결 과정: 프로젝트의 짧은 기간을 고려하여 새로운 기술 도입 대신 기존 스택의 문제 해결에 집중함. next.config의 compiler 옵션에 styledComponents: true를 설정하여, 빌드 시점에 서버에서 생성된 스타일을 클라이언트에 미리 주입하는 방식으로 문제를 해결함.
    - 결과: 팀의 생산성을 유지하면서 SSR 환경의 기술적 문제를 성공적으로 해결함. 이 과정을 통해 향후 Next.js 프로젝트에서는 Tailwind CSS 등 'Zero-Runtime' 방식이 더 유리하다는 기술적 결론을 내림.
